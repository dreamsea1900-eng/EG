# 設計端交付流程（狀態一 → 狀態二 → FD 銜接）

> 以 VIP 頁面（`handoff/vip/`）為實際案例整理，可作為團隊報告與後續專案的參考範本。

## 流程定義

| 階段 | 內容 | 負責方 |
|---|---|---|
| 接收規格 | 取得頁面需求，於 Figma 繪製介面 | 設計端 |
| 狀態一 | 將介面元素綁定 Design System Token（顏色／字級／間距／圓角） | 設計端 |
| 狀態二 | 產出符合 shadcn/ui 規範的 HTML 程式碼，交付工程端 | 設計端 |
| （狀態二之後）| HTML → TSX 轉換、接真實 API 資料、套用工程慣例 | FD（工程端）|

**關鍵分界線**：狀態二的終點是「產出 HTML」，不包含 React/TSX 化與真實資料串接。
這兩件事屬於工程端的整合工作，設計端不需要（也不建議）越界代做——
卡片／列表資料目前都是佔位資料，真正的 state 管理、API 串接只有工程端清楚怎麼接，
設計端產出的 TSX 即使語法正確，也無法在沒有實際專案環境下驗證，價值有限。

## VIP 頁面交付內容（`handoff/vip/`）

| 檔案 | 性質 | 是否必要 |
|---|---|---|
| `index.html` | 狀態二核心產出物，可直接用瀏覽器開啟預覽 | 必要 |
| `notes.md` | 待確認事項、RWD 邏輯、技術決策記錄 | 必要（務必跟 `index.html` 一起給）|
| `img/` | 部分裝飾用 SVG 原始素材備份 | 補充參考，非必要依賴（`index.html` 已內嵌全部內容，不引用此資料夾）|

## 技術案例：手機版裝飾外框的 9-slice 處理

這是本次專案裡一個值得展示的問題解決案例：

**問題**：手機版（< 1024px）卡片寬度範圍很大（343px ~ 991px），但裝飾外框素材是針對
固定比例設計的。整張等比縮放會讓裝飾框在窄螢幕下被壓得太小；整張裁切滿版則會把圓角
裁出可視範圍外，圓角會失真。

**解法**：採用 9-slice（九宮格縮放）概念——把素材左右各切 48px 當作「固定角」
（圓角與邊框維持原生比例，不被拉伸），中間挖空段以 `preserveAspectRatio="none"`
自由拉伸填滿剩餘寬度。並進一步將素材來源由「桌機版漸層推算值」改為「Figma 手機版
原生匯出的漸層數值」，讓最常見的手機尺寸（~343px）還原度最準確。

**結果**：圓角在任何寬度下都不會因縮放而變形，效果與數據已記錄在
`handoff/vip/notes.md`「設計還原度修正（第五輪）」。

## 交付前 Checklist

- [ ] `notes.md` 的「待確認事項」是否都已跟工程端／PM 對齊
      （例如本案的 VIP2~5 數值估讀、`text-button-primary-01` token 對應疑問）
- [ ] Figma asset 暫存連結（`figma.com/api/mcp/asset/...`）是否已下載到專案 `/public`
      （**7 天後失效**，正式交付前需重新確認）
- [ ] RWD 斷點邏輯是否與工程端的 layout padding 機制核對過
      （避免 padding 疊加問題，本案已記錄在 `notes.md`「與工程 Layout 整合注意」）
- [ ] 是否已說明清楚「狀態二只到 HTML，TSX 轉換由 FD 接手」，
      避免被誤解為設計端要產出可運行的 React 元件
- [ ] 座標／尺寸數值是否以 Figma `get_metadata` 精確核對，而非僅憑縮圖視覺估讀
      （多個重複元件時，標記清楚哪些已實際抓取確認、哪些為估讀待確認）
- [ ] 方向性屬性（填充方向、對齊方向等）是否已核對程式碼邏輯，避免肉眼誤判
      （尤其 RTL 原稿可能造成視覺錯覺，例如看起來置左其實程式碼是置右）
